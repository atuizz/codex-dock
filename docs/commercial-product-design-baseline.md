# Codex Dock 商业产品设计与实施基准

更新日期：2026-05-25

## 审查结论

当前仓库已经具备可继续工程化的骨架，而不是空白原型：

- 云控制台已有账号池、OAuth 回调自动接收、RT-only 可用性判断、切换进度、Helper 诊断、审计和管理员界面。
- Worker 已拆分为认证、账号、用户设置、Helper、审计和管理员模块，D1 保存加密 auth 与设备 token，入口已有 request id 和结构化日志。
- Helper 已包含本机 auth 写入、Codex 重启、OAuth 回调、本地状态、自动切换监视和初步定制窗口/托盘 UI。
- 现有静态模块与 Worker 验证脚本、Static Assets 构建在本次审查起点可通过。

距离正式运营仍存在发布阻断项：

1. `docs/usage-refresh-channel-strategy.md` 描述的四种额度刷新通道尚未成为设置、API 和实际刷新闭环；当前网页仍把 Helper 在线当作刷新前置条件。
2. `docs/auto-switch-task-continuity-strategy.md` 的“触发与执行分离”仅有局部骨架，仍需证明低额度和耗尽信号不会在在途轮次内改写 auth。
3. Helper 的新 `RichTextBox` 日志界面尚未具备渲染异常隔离、持久日志恢复与窗口收起/恢复回归证据，不能宣布桌面稳定。
4. 发布工程仍缺统一 CI/CD、预览/生产门槛、Helper 可分发版本清单与可重复 E2E 证据。

## 实施顺序

1. 将额度刷新策略做成完整业务闭环：设置持久化、来源标记、Helper/Cloud/Auto/Manual 执行、有限速批量刷新和精准审计。
2. 收敛 Helper 日志/窗口生命周期，并把自动切换安全边界变成可测试状态机与可读状态。
3. 用工作台、账号导入/详情/管理、Helper 三组视觉基线统一网页与桌面 UI，并进行真实截图回验。
4. 补齐 CI/CD、部署/升级文档、版本分发和线上 smoke 流程后再发布。

## 视觉与交互基线

视觉命题：Codex Dock 是一块安静、可信的运行控制台，以浅灰工作面、深墨文字、低饱和蓝色动作和明确的琥珀/红色风险反馈承载长时间操作。

信息架构：首屏直接显示可操作工作台；主导航固定为概览、账号健康、导入与授权、智能切换、设备与 Helper、审计、设置，并仅向管理员显示管理入口。

交互命题：

- 状态变化采用短促的颜色/文案过渡和进度反馈，不让用户猜测系统是否在工作。
- 导入、切换、刷新、授权撤销等关键动作提供单一主按钮、阶段状态和明确失败下一步。
- 低额度与任务保护显示“正在观察/等待边界/已安全切换”的业务阶段，而不是泛化的错误标签。

## 原型覆盖要求

| 原型 | 必须表达的真实功能 | 实现回验重点 |
| --- | --- | --- |
| 云控制台工作台 | 账号健康列表、剩余额度、实际刷新来源、任务保护提示、诊断日志、Helper 在线状态 | 桌面/平板/手机布局、筛选与状态标记、刷新来源可读 |
| 账号与运营流程 | OAuth 登录导入主路径、账号可用性诊断、刷新方式设置、管理员运营指标/用户治理 | 弹窗层级、失败下一步、技术字段降级、权限入口 |
| Windows Helper | 主窗口状态、分阶段任务弹窗、精简托盘菜单、持久日志与安全边界状态 | DPI、窗口关闭/恢复、日志持续、托盘动作与进度边界 |

## 发布阻断证据

- 自动验证覆盖四种刷新策略、刷新来源和限速/审计去噪。
- 自动验证及真实交互覆盖任务保护边界与手动强制切换警告。
- 构建后的 Helper 真实执行关闭/托盘恢复/重启服务/日志恢复矩阵，不出现异常弹窗。
- 本地浏览器多视口截图与部署后 smoke 证明资源/API/认证路径有效且响应不暴露密钥。
