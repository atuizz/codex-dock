# Codex Dock 商业化问题 Backlog

本文档归纳商业化运营水准的持续改进项。核心商业化批次已转入发布验收，当前发布检查以 [release-and-verification.md](release-and-verification.md) 为准。

## 当前状态

- RT-only 主策略已落地：默认只有带有效 `refresh_token` 的账号可用于 Codex 切换、智能切换和 Helper 自动切换。
- AT-only 账号已明确标注“不支持 Codex”，默认不进入候选。
- OAuth 回调基础闭环已验收成功：用户已通过回调流程成功导入一个账号。
- Helper、Worker、前端已具备基本联动：回调接收、账号导入、账号池展示、手动切换、自动切换框架、审计和本地验证脚本都已经存在。
- 当前整体已具备商业化运行骨架：账号体系、设备授权、RT-first 导入、额度刷新通道、任务连续性保护、Helper 日志可靠性、管理员用户/设备/审计视图、账号健康/失败趋势/Helper 版本聚合指标、内容哈希静态资源版本戳和 CI/CD 基线均已落地；剩余事项主要是更多真实账号场景回归和 Helper 在线升级分发。

## 当前优先级：账号回调自动读取

目标是把“登录导入 RT 账号”打磨成普通用户能稳定完成的主路径，尽量隐藏 OAuth、code、state、PKCE 等技术细节。

### 已完成

- Helper 监听 `http://localhost:1455/auth/callback`，浏览器回调页不再直接报错。
- 回调页通过 `postMessage` 主动通知控制台，控制台保留轮询 Helper 的兜底能力。
- 前端支持从完整回调地址和裸 `code=...&state=...` 中解析授权结果。
- OAuth PKCE 已按 `state` 保存最近记录，降低多次打开授权页时互相覆盖的风险。
- 静态资源已加版本戳，减少线上旧 `app.js` 缓存导致的新旧逻辑混用。
- OAuth 回调解析已抽离到 `oauth-core.js`，旧 state / 错 state / 缺 state 会被识别为不可归属回调，页面只给“重新打开授权页面”这一条主动作。
- 导入弹窗已把“登录导入 RT 账号”设为唯一推荐主入口；Session JSON、当前设备和批量文件降级到高级折叠区。

### 仍需打磨

- 新鲜回调多次重复导入验证：同一浏览器、多浏览器、刷新页面、关闭再打开、失败后重试。
- 旧 code 已过期或已被使用时，还需要更多真实 OpenAI 返回文案样本来继续压缩错误提示。
- 自动读取状态展示：导入弹窗已显示“等待授权”“已收到回调”“正在换取 RT”“导入成功/失败”，后续继续观察真实账号导入场景。
- 自动读取失败兜底：手动粘贴已收进备用折叠区，后续继续降低失败前的手动入口存在感。
- 缓存策略制度化：每次前端资源更新自动生成内容哈希版本戳，`asset-manifest.json` 和 `scripts/verify-static-asset-versioning.mjs` 会校验构建产物。

### 验收标准

- 用户点击“打开授权页面”后，完成登录即可自动回到控制台并展示导入预览。
- 成功导入后账号状态直接进入 `rt_ready`，可用于 Codex。
- 失败时只出现一个主错误，并给出唯一下一步，不出现“没有 token 或 code”这类误导错误。
- 重试时不会被旧回调、旧 state、旧页面缓存干扰。

## 后续商业化问题

### 1. 账号健康中心

问题：账号池现在能展示状态，但还不是商业化的“健康中心”。大量 AT-only、低额度、冷却、失效账号混在一起时，用户很难判断先处理什么。

目标：
- 把账号统一分成可用 RT、缺 RT、RT 失效、额度低、冷却中、当前使用、Helper 不可操作。
- 每类状态给出清晰原因和唯一主操作。
- 支持批量筛选、批量处理和批量清理不可用账号。

验收：
- 用户不需要理解内部字段，也能知道每个账号“能不能用、为什么、下一步做什么”。

### 2. 导入弹窗产品化

问题：导入入口仍带有 OAuth、Session JSON、当前设备、上传文件等技术概念，容易让用户选错路径。

目标：
- 默认主入口为“登录导入 RT 账号”。
- Session JSON 明确降级为“仅登记账号，不可用于 Codex”。
- 上传文件保留为高级批量导入。
- 当前设备导入只用于导入当前 Codex 已登录账号，不用于修复其它账号。

验收：
- 新用户看到导入弹窗时，第一反应是点主按钮登录，而不是理解 token 类型。

### 3. 账号详情页

问题：详情页仍偏技术字段展示，商业化版本应把诊断结论前置。

目标：
- 顶部显示账号可用性结论：可用、不可用、需更新、冷却中。
- 不可用时给出具体原因：缺 RT、RT 失效、AT 不支持、额度低、Helper 离线。
- 技术字段折叠到次级区域。
- “更新”按钮只引导导入该账号自己的 RT，不引导拿当前本机 auth 覆盖其它账号。

验收：
- 账号详情页第一屏能回答：这个账号现在能不能切，不能切该做什么。

### 4. Helper 状态诊断

问题：Helper 在线、已授权、Codex 是否空闲、最近一次切换失败原因等信息分散，用户不知道问题在云端、Helper 还是 Codex。

目标：
- 增加 Helper 状态诊断卡片。
- 展示在线状态、设备授权、当前 Codex 状态、最近一次心跳、最近一次切换、最近一次失败原因。
- 提供“重新授权 Helper”“查看本机日志”“刷新状态”等操作。

验收：
- 用户看到 Helper 诊断后，能判断是否需要重启 Helper、重新授权、等待 Codex 空闲或处理账号。

### 5. 自动切换安全边界

问题：自动切换框架已经存在，但商业化运营需要更强的防误切、防连续失败和失败回滚体验。

目标：
- 切换前稳定确认 Codex 当前任务状态。
- 失败后清楚标记失败阶段：候选选择、payload 下发、写 auth、重启 Codex、恢复窗口。
- 连续失败时自动暂停该设备或该账号，避免重复消耗。
- 冷却策略和当前账号避让策略在 UI 中可解释。

验收：
- 自动切换失败不会让用户不知道当前 Codex 登录态处于什么状态。

### 6. 管理后台和运营审计

状态：核心运营摘要已落地，后续可继续细化按用户/设备下钻和更长周期趋势。

问题：审计已从开发排障扩展到运营摘要，但长期运营还需要更多分组下钻。

已完成：
- 管理后台展示用户数、在线 session、账号总数、RT 可用数、AT-only 数、未刷新数、最近额度刷新失败数。
- `/api/admin/summary` 聚合 Helper 在线/离线、待升级数量、版本分布和 24h 审计失败趋势。
- 管理台 UI 直接展示账号健康、失败趋势和 Helper 版本分布，不暴露 token/auth。

仍需长期打磨：
- 按用户/设备查看最近切换、失败原因、导入成功率。
- 进一步拆分失败来源：OAuth 失败、Helper 离线、无候选、RT 失效、额度不足。

验收：
- 运营人员不用查日志，也能知道平台整体健康状况和用户失败集中点。

### 7. Helper 分发和升级

问题：Helper 目前可用，但商业化需要版本管理和自我诊断。

目标：
- Helper 上报版本号、构建时间和能力标记。
- 控制台提示 Helper 是否过旧。
- 支持下载最新 Helper。
- 支持导出本机日志用于客服排查。

验收：
- 用户 Helper 过旧时，控制台能明确提示升级，而不是表现为功能异常。

### 8. E2E 验收和 CI

问题：当前有本地 verification 脚本，但缺少完整端到端商业验收。

目标：
- 建立固定验收流程：新账号 RT 导入、AT-only 导入、缺 RT 账号处理、手动切换、自动切换、Helper 离线恢复。
- 把前端模块和 Worker 单测接入统一命令。
- 每次发布前生成验收记录。

验收：
- 关键商业路径不是靠临时手测判断，而是有可重复清单。

## 推荐处理顺序

1. 完成账号回调自动读取的多场景验收和 UI 状态打磨。
2. 做账号健康中心和账号详情页诊断。
3. 做 Helper 状态诊断和失败阶段展示。
4. 加强自动切换安全边界。
5. 补运营后台指标和审计聚合。
6. 建立 Helper 版本升级与 E2E 发布验收。

## 当前判断

当前项目已经具备商业化的核心技术骨架，但还不是商业化成品。短期最重要的是把“账号导入 RT”从可用打磨到稳定、少解释、少误操作；只有这个入口稳定，后面的账号健康、自动切换和运营后台才有可靠数据基础。
