# Codex Dock 商业产品化最终目标提示词

下面内容可直接作为 `/goal` 目标使用。总体目标保持产品化、商业化与工程交付为中心；三个专项规格仅作为必须满足的质量门槛，不在总目标中展开实现细节。

```text
/goal

请在当前仓库 C:\Users\Administrator\Documents\codex plus 中，将 Codex Dock 云控制台网页与 Windows Codex Dock Helper exe 完整升级为可以正式运营、持续部署、可维护、可验证的商业产品。你需要直接审查现状、设计方案、生成视觉原型、实施代码、补测试、验证真实交互并在条件具备时完成 Cloudflare 部署与 GitHub 工程化，不要只输出建议或半成品页面。

一、核心目标与边界

- 产品包含 Cloudflare Worker + Static Assets 云控制台、D1 数据/API、安全凭据管理、Windows Helper、账号导入与 OAuth、RT 可用性管理、额度状态、手动/智能切换、设备授权、审计、用户登录注册和管理员能力。
- 最终体验应达到正式商业工具水平：清晰、稳定、可信、可运营、可排障、可持续发布。
- 保持 Cloudflare 适配：遵守 Workers、D1 与静态资源的性能和安全约束，避免不必要的重前端框架、无界查询、无界轮询或高成本后台任务。
- 尊重当前工作区已有变更，先审查再修改，不覆盖不属于本目标的已有工作。

二、工作方式

- 先盘点现有架构、功能、页面、Helper 行为、数据结构、部署方式、测试和已知缺陷，形成简洁的产品缺口清单与实施顺序。
- 对交互架构、视觉系统和关键流程先做原型评审，再实现，不进行无依据的零散美化。
- 必须使用 Image2/imagegen 能力生成与产品真实功能一致的高保真原型参考图，至少覆盖：
  1. 云控制台主工作台、账号健康与日志/诊断区。
  2. 账号详情、导入/OAuth、设置、管理后台中的关键弹窗或视图。
  3. Windows Helper 主窗口、任务进度弹窗、托盘右键菜单与状态反馈。
- 原型只能作为设计验证和实施依据，最终必须实现为真实可用界面并用截图回验。

三、产品信息架构与功能完整性

- 云控制台第一屏就是可操作工作台，不做空洞营销落地页。
- 导航与功能分区围绕用户实际工作：概览、账号池/健康、导入与授权、智能切换、设备/Helper、审计日志、设置；管理员独立管理入口。
- 建立登录、注册、退出、会话安全、角色权限、管理员用户治理等正式账户体系。
- 账号导入以 OAuth/RT 可用主流程为首选，普通用户无需理解 token 技术细节；AT-only、RT 失效、额度低、冷却中、Helper 不可用等状态必须可诊断且给出清晰动作。
- 账号详情回答三个问题：能否使用、为何不能、下一步做什么；技术信息放在次级区域。
- 智能切换、手动切换、失败恢复、候选排除、设备授权与撤销均需形成完整业务闭环和准确审计。
- 管理后台应能查看用户、设备、账号健康、失败趋势、Helper 版本与运营所需聚合指标。

四、统一商业级 UI 规范

- 建立可落地的设计令牌：字体层级、文字色/状态色、背景与面板、边框、圆角、阴影、间距、图标、按钮层级、焦点态、禁用态、危险态和加载态。
- 整体风格应克制、精致、可扫描，适合长期操作型工具；不做过度装饰、不使用粗糙系统默认控件、不堆砌无用卡片。
- 云端页面与 exe 在品牌、颜色、圆角、边框、按钮反馈、弹窗、状态标记、空状态、错误提示上统一一致，但尊重桌面端原生交互习惯。
- 主按钮、次按钮、图标按钮、切换开关、标签、表格/列表、弹窗、菜单、日志与状态卡必须具有清晰边界、hover/focus/pressed/disabled 反馈和可访问性。
- Helper 的标题栏、窗口 X、任务弹窗、进度条、托盘图标与右键菜单必须精细、圆润、边界完整，不能出现黑边、裁切、文字挤压或生硬系统风格。
- 托盘右键菜单只保留高频必要动作，例如显示主窗口、打开控制台、重启本地服务、退出助手及精简状态，不将完整页面功能塞入菜单。
- 日志颜色和级别必须表达真实含义：正常、提醒、等待、错误、授权、切换、系统；不得用误导性标题描述失败来源。
- 针对桌面、平板、手机以及 Windows DPI 缩放进行可视验收，任何界面不得出现重叠、溢出、截断或操作失效。

五、三个专项质量门槛

下列专题必须读取对应文档并实现、测试、验收，但在总体工作中保持为独立质量流，不得让主目标偏离产品整体商业化交付：

- 额度刷新执行通道：读取 docs/usage-refresh-channel-strategy.md。支持本机 Helper、云端 Worker、自动选择与仅手动模式；默认推荐本机网络通道；显示实际刷新来源并避免高频审计噪音。
- 自动切换时机与任务连续性：读取 docs/auto-switch-task-continuity-strategy.md。额度接近或显示耗尽时，不能误切正在继续执行的当前轮；只有确认安全轮次边界后才自动切换，保护最后可用额度并提供可解释状态。
- Helper 窗口与日志可靠性：读取 docs/helper-window-lifecycle-and-log-reliability.md。解决关闭/收起窗口出现 .NET/RichTextBox 异常和日志消失的问题，使日志持久、可恢复且桌面生命周期稳定。

六、架构、安全与性能

- 梳理前端模块、Worker 路由/服务、D1 schema/迁移、Helper 模块边界，按真实复杂度做必要重构，减少巨型文件与重复状态判断。
- 凭据和敏感信息必须安全处理：列表和审计不暴露 token/auth；Worker secret、D1 加密、设备 token 生命周期、撤销与最小权限需要有明确边界。
- API 返回统一结构、错误码、request id 和可诊断摘要；审计既支持用户理解，也支持运营追踪。
- 高频检查、批量刷新、日志和 D1 写入必须限速、聚合、可取消、可观察，符合 Cloudflare 成本与运行限制。
- Helper 本地 API 只在必要边界监听；本地日志与诊断导出必须脱敏。

七、GitHub、CI/CD 与发布能力

- 将项目整理为清晰的 GitHub 仓库工程，补齐 README、架构说明、配置说明、环境变量/密钥说明、部署说明、变更与发布流程。
- 配置自动化 CI：前端/Worker 静态检查与测试、Helper 构建验证、关键业务测试、发布前验收命令。
- 配置可控 CD：Cloudflare Worker/Static Assets 部署流程，区分预览与生产，敏感配置走 secrets，不写入仓库。
- 为 Windows Helper 建立版本号、构建产物、下载分发与升级提醒路径，控制台能识别版本过旧。
- 在有权限且验证通过时完成部署或提交发布所需变更，并记录线上验证证据。

八、验证和完成定义

- 所有新增/修改功能必须有与风险匹配的自动测试或验证脚本，并执行通过。
- 使用真实浏览器回验云控制台关键流程及多视口截图；使用真实构建 exe 回验主窗、托盘、弹窗、日志、关闭/恢复与本地健康接口。
- 对登录注册、OAuth/RT 导入、额度刷新、手动切换、智能切换、任务保护、Helper 离线/重连、审计与管理员路径做端到端验收。
- 对 Cloudflare 部署后的线上静态资源/API/登录流程做 smoke test，确认没有 token 泄露、缓存错版或明显性能回退。
- 最终交付需说明完成内容、架构变化、可见原型与截图、测试结果、部署/CI 状态、剩余风险及后续运营建议。

九、硬性验收标准

- 用户可注册登录并稳定管理自己的账号、设备与设置。
- 云控制台和 Helper 视觉、状态语义与交互均达到统一、精细、可长期使用的商业产品水准。
- Helper 窗口、托盘菜单、任务弹窗和日志没有已知崩溃、异常弹框、边界绘制缺陷或日志丢失。
- 额度刷新通道可配置、可解释、有限速且不污染审计。
- 自动切换不打断仍能继续运行的当前任务，不浪费账号尾部可用额度，并在安全边界完成可靠切换。
- 账号、凭据、设备授权、审计、安全与运营后台形成闭环。
- 仓库具备测试、CI/CD、Cloudflare 发布和 Helper 分发更新的正式工程能力。

执行过程中持续推进实现、验证和修正，除非遇到无法自行解决的外部授权或产品决策阻断，不要停留在提案阶段。
```
