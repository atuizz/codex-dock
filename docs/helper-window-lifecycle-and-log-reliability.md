# Helper 窗口生命周期与日志可靠性治理

## 文档定位

本文档只处理 Windows Helper 的窗口关闭、托盘收起、任务弹窗关闭和日志显示可靠性问题。它属于桌面端稳定性专项，不应让总体商业化目标退化为局部 WinForms 修补任务。

## 已观察到的问题

用户在关闭 Helper 窗口时观察到：

- 出现 Microsoft .NET Framework 未处理异常弹窗。
- 异常信息包含 `System.ArgumentException: 文件格式无效。`
- 堆栈指向 `System.Windows.Forms.RichTextBox.StreamIn`、`RichTextBox.set_Rtf`、`RichTextBox.AppendText` 以及 Helper 的日志追加路径。
- 异常后重新查看主窗口，执行日志区域为空。
- 点击重启服务后，新的日志输出才重新出现。

这不是可接受的商业桌面软件行为：窗口收起到托盘本应是无感且可逆的动作，绝不能导致异常对话框或排障日志消失。

## 当前代码关联路径

当前可确认的相关实现位于 `native-helper/CodexPlusLocalHelper.cs`：

- `MainForm_FormClosing(...)` 在用户点击主窗口关闭按钮时取消关闭并调用 `HideToTray()`。
- 窗口隐藏后，本地服务、状态监控和自动切换服务继续运行，仍可能产生日志。
- `Log(...)` 先写入本地日志文件，再通过 `AppendLogText(...)` 将带颜色的片段追加到 `_logBox`。
- `_logBox` 是 WinForms `RichTextBox`，富文本追加失败时目前缺少渲染隔离和可见恢复机制。
- 任务进度小弹窗也存在独立的 `Close()` 生命周期，需要与主窗口隐藏、真实退出区分处理。

异常栈说明富文本控件路径与故障强相关，但现阶段不能武断认定唯一根因。需要区分以下可能性：

- 窗口隐藏、句柄变化或退出过程中，后台日志回调仍写入 RichTextBox。
- RichTextBox 的内部 RTF 状态被某次富文本追加破坏，下一次 `AppendText` 在内部 `StreamIn` 时报错。
- 关闭主窗、关闭任务弹窗、状态监控刷新或服务重启的 UI 回调发生时序竞争。
- 某类日志内容、字体或选择状态在当前 WinForms/.NET 环境中触发富文本控件异常。

## 目标

- 点击主窗口 `X` 时稳定收起到托盘，不出现任何异常弹窗。
- 从托盘重新打开窗口后，日志仍然存在并持续更新。
- 真正退出应用时后台服务、定时器、弹窗和托盘图标有序释放。
- 日志数据的可靠性不依赖某个 UI 控件是否可绘制。
- RichTextBox 或其它渲染组件发生异常时，后台业务不停止，用户仍可恢复查看最近日志。

## 诊断与复现计划

### 生命周期埋点

在不依赖 RichTextBox 的纯文本日志中记录以下事件：

- 主窗 `FormClosing` 的 `CloseReason`、是否转为托盘隐藏。
- `Visible`、`IsDisposed`、`IsHandleCreated`、`Disposing` 状态变化。
- 托盘显示主窗、重启服务、退出助手的触发来源。
- 任务弹窗创建、用户关闭、自动关闭、主窗口隐藏时的归属关系。
- UI 日志队列写入、绘制跳过、渲染恢复和捕获的异常摘要。

埋点仅记录生命周期与错误摘要，不保存 auth 或 token。

### 复现矩阵

每种场景需要重复执行并保存结果：

1. Helper 静止时点击主窗口 `X`，再从托盘打开。
2. 日志持续刷新时反复收起和打开主窗口。
3. 自动切换检查、额度刷新或服务重启产生日志时关闭主窗口。
4. 任务进度弹窗可见时关闭弹窗、关闭主窗口、再恢复主窗口。
5. 托盘菜单中执行重启服务与退出助手。
6. 日志含网址、中文、异常文本、特殊字符和大量历史行时反复操作。
7. 高 DPI、多显示器或窗口缩放后重复上述操作。

### 根因判定

修复前必须得到至少一种可稳定复现路径，并确认：

- 触发异常的是哪个 UI 动作和哪个日志追加调用。
- 异常发生前 RichTextBox 是否仍有有效句柄、是否正在隐藏或销毁。
- 本地持久日志在异常期间是否仍然完整。
- 日志区域变空是控件状态损坏、窗口重建未回填，还是服务日志停止写入。

## 推荐架构调整

### 日志数据与渲染解耦

日志文件或内存中的结构化日志缓冲必须是事实源，RichTextBox 只负责显示：

- 后台事件先写入线程安全日志存储，再发出 UI 更新通知。
- UI 日志渲染失败不能抛出到主消息循环，也不能阻断后台服务。
- 主窗口重新显示时，从日志事实源重新装载最近若干行，而不是依赖 RichTextBox 仍保留全部状态。
- 设置内存与显示行数上限，避免长时间运行导致富文本控件负担持续增大。

### 安全 UI 分发

- 所有来自后台线程的 UI 更新进入单一队列，由 UI Timer 或安全调度器批量刷新。
- 调用前检查窗口和控件是否已释放、是否已创建句柄、是否正在真实退出。
- 用户只是收起到托盘时可以继续积累日志；窗口隐藏期间可减少即时绘制，重新打开时补绘。
- 真正退出时先停止新事件入队，再停止服务和定时器，最后释放 UI 与托盘资源。

### 富文本渲染故障恢复

短期可继续使用 RichTextBox，但必须有保护层：

- 对日志渲染捕获 `ArgumentException`、`InvalidOperationException` 与释放期间异常。
- 捕获后记录到纯文本故障日志，重建日志控件或清空其渲染状态，并从安全日志缓冲重绘最近内容。
- 对日志消息采用纯文本追加与分段着色，禁止拼接或注入自造 RTF 字符串。
- 若稳定性测试仍能复现内部 RTF 故障，改为不依赖 RichTextBox RTF 流的自绘日志列表或只读文本渲染组件。

### 窗口行为规范

- 主窗口 `X` 默认语义固定为“收起到托盘”，窗口不销毁，任务不停止。
- 托盘菜单中只保留高频必要入口，并提供明确的“退出助手”作为真实销毁路径。
- 任务进度弹窗关闭不影响主窗口日志，不应成为退出主程序的隐式入口。
- 服务重启只重启本地服务和监控，不应作为恢复 UI 日志的必要操作。

## UI 状态与错误提示

正常产品体验不应向用户暴露 .NET 未处理异常框。

当 UI 日志渲染真的发生故障时：

- Helper 继续运行并保留日志。
- 主窗口展示非阻断提示，例如“日志视图已恢复，完整记录仍保存在本机”。
- 提供“打开日志文件”或“导出诊断包”操作，方便客服排障。
- 诊断包应脱敏，不能包含 auth 内容、refresh token 或 device token。

## 实现落点

### Helper 主窗

- 整理 `MainForm_FormClosing(...)`、`HideToTray()`、`ShowFromTray()`、`ExitApplication()` 的状态转换。
- 增加真实退出标志和 UI 日志刷新生命周期标志，避免关闭过程写控件。
- 在重新显示主窗时触发日志恢复加载和状态刷新。

### 日志模块

- 将文件持久化、日志分类和 UI 渲染从 `Log(...)` 中解耦为明确模块。
- 为最近日志建立有界内存缓冲或从滚动文件读取的恢复方法。
- 日志分类颜色继续保留，但颜色渲染不得影响文本可靠性。

### 任务弹窗与托盘

- 验证任务弹窗的自动关闭 Timer 在弹窗、主窗或应用退出时不会回调已释放对象。
- 验证托盘右键菜单执行重启/退出时不会与日志刷新竞争。
- 保证任务弹窗、主窗和托盘使用统一窗口生命周期语义。

## 自动化与人工验证

### 自动化验证

- 对日志缓冲、分类和重载行为做单元测试。
- 对关闭/隐藏/显示/退出状态机做可测试封装，覆盖后台日志写入期间的行为。
- 为包含中文、网址、括号、反斜线、大批量文本的日志渲染添加回归测试。
- 构建后执行 Helper 健康接口验证，确认 UI 异常不会影响本地 API。

### 人工验收

1. 连续运行 Helper 并产生实时日志，收起和打开主窗至少 `30` 次，无异常。
2. 任务弹窗存在和自动关闭期间执行主窗收起、托盘打开、服务重启，无异常。
3. 托盘退出可一次干净退出，无残留进程、无错误弹窗。
4. 重开主窗口后，关闭前后的日志连续可见，不需要重启服务恢复。
5. 日志达到显示上限后滚动与颜色分级正常，窗口交互仍流畅。

## 验收标准

- 任何普通关闭、收起、重开、重启服务和退出流程均不再出现 `RichTextBox` 或 `.NET Framework` 未处理异常。
- 窗口收起和恢复不会清空执行日志。
- 即使 UI 渲染被人为注入异常，本地服务和持久日志仍继续工作，界面能够自恢复。
- 错误文案对用户简洁明确，对诊断提供足够证据，同时不泄露敏感凭证。
- Helper 窗口、任务弹窗、托盘菜单的生命周期行为一致、可预测、可重复验收。
